## 2. Vercel — Deployment Configuration

### What Vercel Runs

Vercel deploys the Next.js application from the GitHub `main` branch. It serves:

- **Pages:** SSG/SSR pages (event pages, dashboard, landing)
- **API routes:** All `app/api/*` endpoints
- **Static assets:** `/public/*`
- **Cron jobs:** `/api/cron/process-jobs` runs every minute

### Required Vercel Configuration

**`vercel.json`** — needs to be created:

```json
{
  "framework": "nextjs",
  "regions": ["gru1"],
  "crons": [
    {
      "path": "/api/cron/process-jobs",
      "schedule": "*/1 * * * *"
    }
  ]
}
```

### Environment Variables (set in Vercel Dashboard)

| Variable | Type | Source |
|----------|------|--------|
| `NEXT_PUBLIC_SUPABASE_URL` | Public | Supabase project settings |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Public | Supabase project settings |
| `SUPABASE_SERVICE_ROLE_KEY` | Secret | Supabase project settings |
| `ABACATEPAY_API_KEY` | Secret | AbacatePay dashboard |
| `ABACATEPAY_WEBHOOK_SECRET` | Secret | AbacatePay dashboard |
| `RESEND_API_KEY` | Secret | Resend dashboard |
| `NEXT_PUBLIC_APP_URL` | Public | Your domain (e.g. `https://ticket.vercel.app`) |
| `JOB_PROCESSOR_SECRET` | Secret | Random UUID generated by you |

### Vercel Setup Steps

1. Push code to GitHub
2. Import repo in Vercel
3. Configure environment variables
4. Deploy (triggers automatically on `main` push)
5. Configure custom domain (or use `*.vercel.app`)
6. Set AbacatePay webhook to `https://your-domain.com/api/webhooks/abacatepay`

## 3. Supabase — Database & Auth

### What Supabase Provides

- **PostgreSQL database** — schema via migrations
- **Auth** — email/password sign-up, session management
- **Storage** — `event-covers` bucket for event images

### Migration Files

```
supabase/migrations/
  00001_initial_schema.sql   — All tables (organizers, events, tiers, orders, etc.) + indexes
  00002_functions.sql        — PostgreSQL functions (job processing, etc.)
  00003_atomic_checkout.sql  — Atomic checkout functions with row-level locking
```

### Supabase Setup Steps

1. Create Supabase project in **southamerica-east1** (São Paulo)
2. Enable email/password auth provider
3. Create `event-covers` Storage bucket (public)
4. Apply migrations via Supabase CLI or SQL editor
5. Copy API keys (URL, anon key, service_role key) to Vercel env vars

### Supabase Project Configuration

| Setting | Value |
|---------|-------|
| Region | `southamerica-east1` (São Paulo) |
| Auth providers | Email/Password enabled |
| Storage buckets | `event-covers` (public read, authenticated write) |
| Row Level Security | Not used — access control is in API route code |
| Connection pooling | pgBouncer (enabled by default) |

### Schema Summary

```
organizers     → events     → tiers
                           → orders → order_items → tickets → check_ins
                           → pending_jobs
```

## 4. CI/CD Pipeline (GitHub Actions)

### File to Create: `.github/workflows/ci.yml`

Triggers on every push (all branches). Steps:

1. Checkout code
2. Setup Node.js
3. Install dependencies (`npm ci`)
4. Run TypeScript type check (`npm run build` or `tsc --noEmit`)
5. Run tests (`npm test`)

**Note:** Tests require env vars. On CI (push to PRs before merge), tests that hit external services (Supabase, AbacatePay, Resend) should be mocked. The CI env vars from GitHub Secrets provide the real credentials only on `main`.

## 6. Deployment Order

This is the sequence you should follow to go from code to live system:

### Phase A — Supabase Setup (do first)

```
1. Create Supabase project (southamerica-east1)
2. Enable email/password auth
3. Create event-covers storage bucket (public)
4. Run all 3 migrations in order (00001 → 00002 → 00003)
5. Save API keys (URL, anon key, service_role key)
```

### Phase B — GitHub + CI Setup

```
6. Create GitHub repository
7. Create required files:
   - vercel.json
   - .github/workflows/ci.yml
   - README.md
8. Push code
9. Configure GitHub Secrets for CI:
   - NEXT_PUBLIC_SUPABASE_URL
   - NEXT_PUBLIC_SUPABASE_ANON_KEY
   - SUPABASE_SERVICE_ROLE_KEY
   - ABACATEPAY_API_KEY
   - RESEND_API_KEY
   - NEXT_PUBLIC_APP_URL
   - CRON_SECRET
10. Verify CI passes on push
```

### Phase C — Vercel Deploy

```
11. Import repo in Vercel
12. Configure env vars (same set as GitHub Secrets)
13. Deploy
14. Verify deployment at *.vercel.app domain
15. Configure Cron Job (vercel.json handles this)
```

### Phase D — External Services

```
16. Create AbacatePay account
17. Set AbacatePay webhook → https://your-domain.com/api/webhooks/abacatepay
18. Create Resend account
19. Verify domain in Resend (SPF/DKIM setup)
20. Set AbacatePay webhook secret in Vercel env vars
```

### Phase E — LGPD Compliance

```
21. Create app/privacy/page.tsx
22. Create app/api/admin/delete-attendee-data/route.ts
23. Verify privacy page accessible at /privacy
```

### Phase F — Launch

```
24. Complete E2E walkthrough (M7 checklist)
25. Run full test suite
26. If everything passes → ship
```

## 8. Risk & Gotchas

### Common Deployment Issues

| Issue | Mitigation |
|-------|-----------|
| **Vercel 10s function timeout** | Webhook handler returns immediately, defers work to `pending_jobs` queue |
| **Vercel free tier limits** | 100h serverless execution / month — monitor usage |
| **Supabase free tier** | 500MB DB, 50K MAU auth — sufficient for MVP |
| **Migrations out of sync** | Always apply migrations in order, never skip |
| **Missing env vars** | `lib/env.ts` throws in production on missing vars — check Vercel dashboard |
| **Cron Job fails** | Check Vercel Cron logs; ensure `JOB_PROCESSOR_SECRET` matches |
| **AbacatePay webhook not reaching** | Use a public URL (not `localhost`); check webhook logs in AbacatePay dashboard |
| **Resend emails bouncing** | Verify domain SPF/DKIM; check Resend logs |

### Files That Must NOT Be Committed

```
.env.local
.env.*.local
*.pem
.next/
node_modules/
coverage/

.vercel/

```
